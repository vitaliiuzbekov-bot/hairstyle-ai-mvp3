# Текущее состояние проекта (НейроСтилист AI)

## Что было сделано:
1. **Биллинг и Ошибка Авторизации**: Исправлена логика в `billing.ts`. Ранее пользователи, у которых в базе не был записан `tgId` (созданные до введения обязательной привязки к Telegram), получали `AUTHORIZATION_ERROR`, из-за чего интерфейс писал "не хватает кредитов/звезд" даже если на балансе было 999. Теперь старым пользователям автоматически прописывается `tgId`.
2. **Ежедневный Бонус**: Исправлен баг, из-за которого ежедневный бонус (+1 генерация) добавлялся только визуально в локальное хранилище браузера, но не синхронизировался с сервером Firebase. Теперь сервер получает токен.
3. **Камера в Telegram**: В `usePhotoHandlers.ts` убрана жесткая проверка мобильного устройства в Telegram, из-за которой кнопка "Сделать фото" принудительно открывала файловый менеджер (который на iOS мог открывать только галерею). Теперь приложение сначала пытается запросить встроенную камеру через `navigator.mediaDevices.getUserMedia`.
4. **Фотографии Топ Стрижек**: Решена проблема с неотображением иконок топовых стрижек после экспорта. Ранее они зависели от локальных файлов `m_french_crop.jpg` и т.д., которые вырезались при определенной конфигурации сборки Vite. Компонент `PresetsCarousel` полностью переведен на использование `LazyImage`, который запрашивает картинки из официальной библиотеки через серверный API. Названия стрижек (например, "French Crop") жестко синхронизированы с базой (пункт 8), поэтому загрузка происходит из кэша мгновенно.

5. **Экспорт в PDF**: Картинка исходного фото теперь загружается через специальный прокси (`/api/proxy-image`) для обхода CORS ограничений, поэтому она гарантированно попадает в PDF-файл.

6. **Ошибки старых версий Telegram**: В компоненте добавления на главный экран (`PWAPrompt`) добавлена строгая проверка версии Telegram (`tg.isVersionAtLeast("8.0")`). Это предотвращает падение приложения (unsupported method error) у пользователей с устаревшими клиентами Telegram, где функция `checkHomeScreenStatus` отсутствует.

7. **Исправление ошибок загрузки чанков**: Добавлен авторефреш при ошибке загрузки динамических чанков, которые могут возникать после деплоя новых версий.

8. **Исправление Nginx 403 HTML**: Убраны ответы с 403 кодом, которые перехватывались nginx, что приводило к ошибке парсинга JSON на клиенте. Теперь сервер отдает 400 Bad Request.

9. **Изображения референсов (Библиотека)**: Названия стрижек в карусели "Топ-стрижек месяца" (например, "Френч кроп") не совпадали с названиями в официальной базе библиотеки (там было "French Crop"). Из-за этого срабатывала защита сервера (возвращая 400/403). Имена полностью синхронизированы, теперь все картинки карусели загружаются корректно.

10. **Баг с 999 генерациями**: У разработчиков в интерфейсе появлялось 999 генераций, но сервер выдавал ошибку "Закончились генерации". Причина заключалась в том, что Nginx-прокси вырезал кастомный заголовок `x-developer-mode`. Теперь статус разработчика передается прямо в теле запроса (FormData), обходя ограничения прокси.

11. **Telegram Stories**: Исправлена ошибка "К сожалению, не удалось поделиться в сторис". Проблема была в том, что функция браузера для генерации видео (`canvas.captureStream`) не поддерживается на некоторых версиях iOS/Safari в Telegram. Теперь добавлен безопасный "фолбэк": если создание видео падает с ошибкой, приложение мгновенно создает и отправляет в сторис статичную фотографию в формате До/После (коллаж).

12. **Качество генерации**: Улучшен промпт Gemini (системные инструкции) для предотвращения изменения черт лица у лысых или бритоголовых людей при кардинальной смене прически. Теперь нейросеть получает строгие инструкции не менять изначальное строение головы.

13. **Тестирование Stories**: Улучшена логика предупреждений для тестирования функции "В сторис" вне среды Telegram (например, в песочнице Google Studio). Если кнопка нажимается из обычного браузера, пользователю показывается понятное окно с объяснением.

14. **Сжатие фотографий на клиенте (Оптимизация)**: Параметры сжатия увеличены до 1024x1024 пикселей (качество 80%), как и требовалось. Добавлено принудительное масштабирование кадра (до 1024px) прямо в момент снимка с веб-камеры (`useCamera.ts`), что экономит оперативную память и предотвращает лаги при снимках с 4K-камер телефонов.

15. **Безопасность биллинга (Fail-Closed)**: Устранена уязвимость (Fail-Open), при которой ошибки подключения к Firebase Admin SDK (отсутствие БД или `PERMISSION_DENIED`) возвращали `{ ok: true }`. 16. **Защита SSRF (Проксирование изображений)**: В эндпоинт `/api/proxy-image` добавлена строгая проверка принимаемых URL. Разрешены только домены из белого списка (`fal.media`, `v3.fal.media`, `storage.yandexcloud.net`, `firebasestorage.googleapis.com`, `images.unsplash.com`) и только протокол HTTPS. Это предотвращает использование сервера в качестве открытого прокси.
17. **Защита вебхука Telegram**: Эндпоинт `/api/set-telegram-webhook` в `src/server/routes/auth.ts` защищен обязательной проверкой заголовка `x-admin-secret`. Теперь для его вызова значение заголовка должно совпадать с переменной окружения `ADMIN_SETUP_SECRET`, иначе возвращается 403.
18. **Защита Developer Mode**: В эндпоинтах генерации (`generate.ts` и `reference.ts`) убрана проверка простого заголовка `x-developer-mode` (который мог подделать любой). Теперь используется криптографическая проверка через новый хелпер `isAuthorizedDeveloper` (в `tgAuth.ts`). Он проверяет валидность данных `x-telegram-init-data` (созданных Telegram) и сверяет Telegram ID пользователя со списком разрешенных из переменной `DEV_TELEGRAM_USER_IDS`.
19. **Безопасность Storage (Загрузка фото)**: Обновлены правила Firebase Storage (`storage.rules`). В директории `/uploads/` устранено разрешение `allow write: if true;`, которое позволяло любому желающему загружать любые файлы. Теперь загрузка разрешена только авторизованным пользователям (`request.auth != null`), только изображений (`image/.*`), и размер ограничен 15 МБ.
20. **Исправление ошибок генерации Fal.ai (422 Error)**: Локализован и устранен критический баг загрузки изображений в Fal.ai. Изменение конвертации base64, введенное ранее (использование Buffer), приводило к повреждению бинарных данных изображения при загрузке. Из-за этого модели Fal (`face-swap` и `flux`) возвращали ошибку 422 ("Could not load image from url"). Восстановлен правильный механизм создания Blob через `node-fetch`.
21. **Настройка Webhook Telegram**: Вебхук успешно установлен через защищенный эндпоинт `/api/set-telegram-webhook` с использованием правильного `ADMIN_SETUP_SECRET`.
22. **Обработка ошибок Fal.storage**: Изменена логика в `uploadImageToFal` (в `falClient.ts`). Теперь при ошибке загрузки в `fal.storage.upload` выбрасывается исключение с подробным логированием, а не возвращается base64. В `generate.ts` добавлена проверка: если URL загруженного изображения начинается на `data:` (например, при моковом API ключе), выбрасывается понятная ошибка для пользователя: "Не удалось загрузить изображение для обработки. Попробуйте ещё раз."
23. **Устранение ошибки 422 в FaceSwap**: Добавлена принудительная нормализация изображений в `generate.ts` и `falClient.ts`. Если изображение передается в виде сырой строки base64 (без префикса `data:image/...`), система автоматически подставляет префикс перед загрузкой в `fal.storage.upload` или передачей в параметры модели FaceSwap. Это исправляет ошибку 422 "Could not load image from url", так как Fal ожидает либо валидный data URI, либо URL.
24. **Улучшена логика скачивания изображений**: В `resolveImageToBase64` улучшена обработка ошибок сети (добавлен `node-fetch`), теперь при невозможности скачать изображение сервер возвращает понятную пользователю ошибку, а не пытается передать сломанный контент дальше по пайплайну.
25. **Устранение ошибки import_node_fetch.default is not a function**: Удален импорт `node-fetch` из `generate.ts`. В Node.js 18+ встроен глобальный `fetch`, и ручной импорт конфликтовал при сборке esbuild для CommonJS.
26. **Исправление ошибочной классификации Британки как лысины**: Регулярные выражения в `reference.ts` и `promptGenerator.ts` ошибочно классифицировали "британку" как стрижку налысо из-за подстроки "брит". Это приводило к тому, что для "британки" генерировался шаблон абсолютно лысого человека, на который затем накладывалось лицо пользователя в FaceSwap. Теперь поиск подстроки изменен (убрано `includes("брит")`), чтобы предотвратить ложные срабатывания. Также исправлена логика для "Полубокса" (чтобы не срабатывало правило для "бокс" = ежик).
27. **Исправление конфигурации Storage Bucket**: Изменен `storageBucket` в `firebase-applet-config.json` с `.firebasestorage.app` на классический `.appspot.com`, чтобы решить проблему с ошибкой `404 The specified bucket does not exist`. Ошибка вызвана тем, что по умолчанию Firebase Storage создает bucket в домене `appspot.com`.
28. **Исправление 504 Gateway Timeout (нет результата на слайдере)**: Восстановлен механизм поллинга (polling) для генерации VTON. Ранее процесс был синхронным, из-за чего HTTP-запрос на `/api/generate-full` отваливался по таймауту балансировщика Render (100 секунд). В итоге клиент видел ошибку или пустой слайдер, а сервер успевал завершить работу и отправить результат в Telegram. Теперь запрос инициирует фоновую задачу, возвращает `jobId`, а клиент опрашивает `/api/generate-full/status` каждые 3 секунды.
29. **Сброс кеша**: Изменен префикс ключа кеширования на `v2_`, чтобы старые сгенерированные ошибки (например, лысая Британка) больше не возвращались из кеша Firestore.
30. **Гарантированное сохранение статуса задачи (jobs) при генерации**: В `src/server/routes/generate.ts` блок сохранения статуса `job` (`done` или `error`) в Firestore был перенесен в блок `finally`. Это гарантирует, что даже если на этапе сохранения в Firebase Storage произойдет ошибка и будет выброшено исключение (или возникнет любая другая непредвиденная ошибка в пайплайне), клиент не останется вечно опрашивать `jobId` со статусом `processing`. Добавлено детальное логирование: `[generate-full] Saving job status to jobMap/Firestore, jobId: <ID>`. Также `status: 'processing'` теперь возвращается клиенту в первоначальном ответе при инициации генерации.
31. **Исправление вызова на клиенте для поллинга генерации**: `generateFullApi` в `src/services/api.ts` по-прежнему стучался на старый эндпоинт `/api/generate-full` вместо нового `/api/generate-full/start`, что приводило к ответу 404 от бэкенда и ошибке `HTML Proxy Error, HTTP 404` на клиенте из-за SPA fallback-а. Вызов был обновлен для корректной работы с механизмом опроса задач.

32. **Исправление ложной ошибки HTML Proxy Error, HTTP 404 у пользователей с кэшем**: Ошибка `HTML Proxy Error, HTTP 404` возникала у пользователей со старым кэшем приложения, которые продолжали отправлять запросы на удаленный эндпоинт `POST /api/generate-full`. Так как эндпоинт был удален, запрос проваливался в дефолтный обработчик Express, который возвращал HTML страницу с ошибкой `Cannot POST...`. Так как клиент ожидал JSON, парсер натыкался на `<!DOCTYPE html>` и выкидывал ошибку `HTML Proxy Error`. Для исправления ситуации добавлен fallback для всех `api/*` запросов на возврат JSON `404`, а также восстановлен alias `POST /api/generate-full`, который перенаправляет запросы на новый пайплайн `/generate-full/start`, обеспечивая обратную совместимость для клиентов со старым кэшем.
33. **Замена GET на POST для опроса статуса генерации**: Из-за агрессивного кэширования запросов GET в Telegram Mini App, клиенты часто получали статус `processing` при поллинге (jobId), даже когда сервер уже обработал запрос. Изменен эндпоинт `/api/generate-full/status` с GET на POST, а к URL на клиенте также добавлен параметр `?t=timestamp` для полного отключения кэша со стороны провайдеров.
